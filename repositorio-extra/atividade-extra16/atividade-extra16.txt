=====================================================
    PROJETO FINAL: SISTEMA DE GESTÃO COMERCIAL (ERP)
=====================================================

1. O CONCEITO DE ERP (Enterprise Resource Planning):
Um ERP é o "cérebro" de uma empresa. Ele integra todos
os processos em um único lugar:
- COMPRAS: Gestão de fornecedores e reposição de estoque.
- ESTOQUE: Controle de mercadorias paradas (Capital Imobilizado).
- VENDAS: Interface com o cliente e geração de receita.
- FINANCEIRO: Fluxo de Caixa, Lucro e Pagamento de Contas.

2. LOGICA FINANCEIRA APLICADA:
Neste projeto, lidamos com conceitos reais de mercado:
- CUSTO (CMV): O valor pago ao fornecedor pelo produto.
- MARKUP (Margem): O percentual adicionado para cobrir
  gastos e gerar lucro. (Preço = Custo * (1 + Margem/100))
- GASTO FIXO: Despesas que existem mesmo se você não vender
  nada (Aluguel, Internet, Salários).
- LUCRO LÍQUIDO: O que sobra após pagar o custo do produto
  E os gastos fixos.

3. ARQUITETURA DO SOFTWARE (O QUE USAMOS?):
Integramos absolutamente tudo o que aprendemos:
- VETORES: Para gerenciar o estoque dinâmico de produtos.
- PILHAS: Para criar um LOG de Auditoria (Última ação primeiro).
- FILAS: Para gerenciar a ordem de atendimento de clientes.
- EXCEÇÕES: Para impedir vendas sem estoque ou caixa negativo.
- CORES ANSI: Para criar um Dashboard de Gestão intuitivo.
- MANIPULAÇÃO DE MOEDA: Tudo em centavos para precisão total.

4. ANÁLISE DE FALHA CRÍTICA (RISCOS DE ARQUITETURA):
- Ponto Único de Falha (SPOF): O `SistemaERP` concentra todas as 
  responsabilidades (Estoque, Financeiro, UI, Logs). Se essa classe 
  crescer demais, o sistema se torna um "Monólito Frágil". Recomenda-se 
  dividir em sub-motores (MotorEstoque, MotorFinanceiro) no Nível 21+.
- Inconsistência de Persistência: O arquivo `produtos.txt` é lido 
  no início, mas o sistema não salva as alterações de estoque em 
  tempo real, apenas no fim ou nem isso. Se o programa travar, o 
  estoque físico e o digital divergirão (Falta de COMMIT/ROLLBACK).
- Visibilidade de Dados: Membros privados como `saldoCaixa` e `estoque` 
  estão protegidos, mas o uso de referências constantes no getter 
  ainda permite que alguém mal-intencionado use `const_cast` para 
  burlar as regras de negócio.
