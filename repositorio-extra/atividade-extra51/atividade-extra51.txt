# 📘 Atividade Extra 51: Composição de Classes (Motor Industrial)
STATUS: Padrão de Engenharia de Elite Aplicado

## 🎯 Estudo de Caso
A fábrica "SmartMatrix" precisa agora de um sistema para monitorar seus **Motores Industriais**. Um motor não é apenas um sensor, mas um sistema composto por múltiplos sensores. Se qualquer um desses sensores detectar uma anomalia, o motor inteiro deve ser desligado por segurança.

## 🛠️ Necessidade Técnica
Em vez de colocar toda a lógica de sensores dentro da classe `Motor`, usamos a **Composição (Relationship "Has-A")**. 

## 🧪 ANÁLISE DE FALHA CRÍTICA (NÍVEL DESIGN DE SISTEMAS)
- RISCO DE ACOPLAMENTO RÍGIDO (TIGHT COUPLING): Se a classe `Motor` depender de uma implementação específica de `Sensor`, qualquer mudança no sensor exige recompilar o motor. Solução: Injeção de Dependência ou Interfaces.
- RISCO DE CASCATA DE FALHA: Se o objeto `Sensor` interno falhar ou for corrompido, o objeto `Motor` perderá sua capacidade de autodefesa física, levando a danos estruturais catastróficos.
- RISCO DE CICLO DE VIDA ÓRFÃO: Em composições dinâmicas (ponteiros), deletar o motor sem deletar seus sensores internos causaria vazamento de memória. Como usamos composição estática (stack-based), o RAII protege o sistema automaticamente.

## 📋 Requisitos de Elite
1. Reutilizar a lógica de `Sensor` (Atividade 50) via inclusão modular.
2. Implementar a classe `MotorIndustrial` usando Member Initializer List para compor o sensor.
3. Delegar a validação de segurança para o sensor interno (Princípio da Responsabilidade Única).
4. Interface ANSI para telemetria de motor e status de segurança.
