# 📘 Atividade Extra 54: Polimorfismo e Sobrescrita (Gestão Imobiliária)
STATUS: Padrão de Engenharia de Elite Aplicado

## 🎯 Estudo de Caso
A imobiliária "HouseLogic" precisa de um sistema para calcular o valor total mensal de seus imóveis. Diferentes tipos de imóveis possuem regras de cobrança distintas:
- **Apartamento:** Valor base + Taxa de Condomínio.
- **Casa:** Valor base + Taxa de Manutenção de Jardim.

## 🛠️ Necessidade Técnica
Para resolver isso, usamos o **Polimorfismo (Polymorphism)** e a **Sobrescrita de Métodos (Method Overriding)**.

## 🧪 ANÁLISE DE FALHA CRÍTICA (NÍVEL ARQUITETURA DE REDES)
- RISCO DE DESTRUIÇÃO INCOMPLETA: Se a classe base `Imovel` não possuir um destrutor virtual, deletar um `Apartamento` via ponteiro `Imovel*` liberará apenas a memória da base. Atributos específicos da filha ficarão "órfãos" na RAM, gerando Memory Leaks.
- RISCO DE LATE BINDING OVERHEAD: Cada chamada a um método virtual exige uma consulta à V-Table. Em sistemas com milhões de objetos processados por segundo, esse "pulo" extra pode impactar o throughput de dados.
- RISCO DE IMPRECISÃO FINANCEIRA: Cobranças de aluguel e condomínio usando `double` causam erros de arredondamento. O sistema deve migrar para o padrão de Guardião Financeiro (Centavos Inteiros).

## 📋 Requisitos de Elite
1. Implementação de Polimorfismo Dinâmico via ponteiros e métodos virtuais.
2. Uso obrigatório da medalha Guardião Financeiro (Cálculos em centavos).
3. Proteção RAII com garantia de destruição polimórfica (Destrutor Virtual).
4. Interface ANSI para telemetria de faturamento imobiliário.
