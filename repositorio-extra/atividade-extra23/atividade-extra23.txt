ESTUDO DE CASO: GATEWAY DE PAGAMENTOS MULTI-MODAL (CHECKOUT)
------------------------------------------------------------------------------
DOMÍNIO: E-commerce / Fintech
ATIVIDADE: Extra 23
CONCEITO NOVO: Classes Abstratas e Métodos Virtuais Puros
STATUS: Padrão de Engenharia de Elite Aplicado

1. O CENÁRIO
Um site de vendas aceita Cartão de Crédito, PIX e Boleto. O sistema de checkout 
possui uma função principal que recebe um "Método de Pagamento" e executa a 
cobrança. 

2. A NECESSIDADE TÉCNICA
Até agora (Atividade 22), podíamos criar um objeto da classe mãe. Mas pense: 
faz sentido existir um objeto do tipo "Pagamento" puro? Não. Todo pagamento 
DEVE ser de algum tipo específico.
A CLASSE ABSTRATA resolve isso:
- Método Virtual Puro: Usamos a sintaxe '= 0'. Isso diz: "Eu não sei como 
  processar isso aqui, mas obrigo minhas filhas a saberem".
- Abstração: Uma classe com pelo menos um método virtual puro não pode ser 
  instanciada (você não pode dar 'new Pagamento()'). Ela serve apenas como 
  um CONTRATO (Interface).

3. 🧪 ANÁLISE DE FALHA CRÍTICA (NÍVEL ARQUITETURA)
- RISCO DE INSTANCIAÇÃO ILEGAL: Tentar criar um objeto de uma classe abstrata 
  causa erro de compilação. Isso protege o sistema de comportamentos indefinidos.
- RISCO DE CHAMADA VIRTUAL NO CONSTRUTOR: Se o construtor da classe base chamar 
  um método virtual puro, o sistema sofrerá um Crash (Pure Virtual Method Call). 
  Nunca chame funções virtuais dentro de construtores/destrutores em C++.
- RISCO DE INTEGRIDADE FINANCEIRA: Usar 'double' para pagamentos pode causar 
  erros de arredondamento de centavos em gateways de alto volume.

4. REQUISITOS DO SISTEMA
- Classe Abstrata 'MetodoPagamento' (Contrato).
- Implementação de 'Guardião Financeiro' (Cálculos em centavos inteiros).
- Uso de Polimorfismo Dinâmico para processar fila de cobrança.
- Interface ANSI para status de transação bancária.
